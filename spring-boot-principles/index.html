🍃 Spring & Spring Boot 核心原理 理清关系 · 深入 IoC · 解密自动装配 · 源码链路

理清关系：Spring Boot 是 Spring 的「一键启动器」

🧩 Spring Framework 一套完整的乐高积木，什么都有，但你需要自己拼
🤖 Spring Boot 一个智能拼装机器人，按你的需求自动拼好积木

Spring Framework 提供了 IoC 容器、AOP、事务管理、MVC 等底层能力，但你需要手动写大量 XML 配置或 @Configuration 类。Spring Boot 在此基础上做了 "约定大于配置" 的自动装配，帮你把常用的组件自动配置好。

Spring Framework 的核心：IoC 容器

什么是 IoC 容器？

IoC（Inversion of Control，控制反转）容器本质上是一个巨大的 Map，key 是 Bean 的名字，value 是对象实例。它负责：

  1. 创建对象（而不是你 new）
  2. 管理对象之间的依赖关系（注入）
  3. 控制对象的生命周期（初始化、销毁）

核心接口是 BeanFactory 和它的增强版 ApplicationContext。

Bean 的生命周期

当你写 @Component、@Service 或 @Bean 时，Spring 容器会经历以下流程：

🔍 扫描类路径 → 📋 解析 BeanDefinition → 🏗️ 实例化 → 🔗 属性填充 → 📢 Aware 回调 → ✅ @PostConstruct → 📦 放入单例池 → ♻️ @PreDestroy
📌 源码入口
这一切的核心入口是 AbstractApplicationContext.refresh() 方法，它内部有 12 个步骤，是整个 Spring 容器启动的"总导演"。

对象的创建和注入

注解一句话标注告诉 Spring
@Bean 往容器里放东西 标注在方法上 这个方法的返回值，帮我创建并管理起来
@Autowired 从容器里取东西 标注在字段、构造器、Setter 方法上 我需要一个 Bean，请帮我找到并塞进来
💡 直观对比
@Bean 就像往仓库里放货品，@Autowired 就像从仓库里取货品。

Spring 三级缓存（解决循环依赖 + AOP）

三级缓存的定义

Spring 三级缓存的源码在 DefaultSingletonBeanRegistry 中，就是三个 Map： 

// 一级缓存：成品 Bean（完全初始化好的）
private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);

// 二级缓存：半成品 Bean（已实例化，未填充属性）
private final Map<String, Object> earlySingletonObjects = new HashMap<>(16);

// 三级缓存：工厂函数（可以生成 Bean 的"图纸"）
private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>(16);

用生活比喻理解三级缓存

缓存级别比喻存储内容
singletonObjects 一级 🏠 精装房，可入住 完全初始化好的 Bean（属性填充完毕，AOP 代理已生成）
earlySingletonObjects 二级 🏗️ 毛坯房，有框架但没装修 已实例化但属性未填充的"半成品"Bean
singletonFactories 三级 📐 建筑图纸 + 施工队 一个工厂函数，可以随时"按图纸造出毛坯房"

为什么需要三级缓存？—— 循环依赖场景

假设有两个 Bean 互相依赖： 

@Component
public class A {
    @Autowired
    private B b;
}

@Component
public class B {
    @Autowired
    private A a;
}

如果没有三级缓存，Spring 会陷入死循环：创建 A → 发现需要 B → 创建 B → 发现需要 A → 创建 A → ...

三级缓存如何解决循环依赖（完整源码链路）

核心源码入口：getBean() → doGetBean() → getSingleton()

Step 1：创建 A 的"图纸"放入三级缓存

// doCreateBean() 中，实例化 A 对象后
// 将 A 的工厂函数放入三级缓存（singletonFactories）
addSingletonFactory(beanName, () -> getEarlyBeanReference(beanName, mbd, bean));

Step 2：创建 B 时发现需要 A，从三级缓存中获取

// getSingleton() 中，从三级缓存获取 A 的工厂函数
protected Object getSingleton(String beanName, boolean allowEarlyReference) {
    Object singletonObject = this.singletonObjects.get(beanName);
    if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
        singletonObject = this.earlySingletonObjects.get(beanName);
        if (singletonObject == null && allowEarlyReference) {
            ObjectFactory<?> singletonFactory = this.singletonFactories.get(beanName);
            if (singletonFactory != null) {
                // 通过工厂函数生成 A 的"毛坯房"（可能已代理）
                singletonObject = singletonFactory.getObject();
                // 放入二级缓存，同时从三级缓存移除
                this.earlySingletonObjects.put(beanName, singletonObject);
                this.singletonFactories.remove(beanName);
            }
        }
    }
    return singletonObject;
}

Step 3：B 拿到 A 的半成品，完成自己的初始化，A 继续完成初始化

✅ 完整流程
  1. 创建 A → 实例化 A → 放入三级缓存
  2. 填充 A 的属性 → 发现需要 B → 创建 B
  3. 实例化 B → 放入三级缓存
  4. 填充 B 的属性 → 发现需要 A → 从三级缓存获取 A（获得 A 的半成品，可能已代理）
  5. B 完成初始化 → 放入一级缓存
  6. A 拿到 B 的成品 → 完成初始化 → 放入一级缓存

为什么是三级？二级够不够？

这是一个经典追问。答案是：如果只有普通 Bean，二级缓存就够了；但 Spring 有 AOP 代理，必须有三级缓存。

🎯 场景：A 需要被 AOP 代理

如果只有二级缓存（直接存原始对象）：

  • B 从二级缓存拿到的是 A 的原始对象
  • 但 A 最终需要的是AOP 代理对象
  • B 持有的原始对象和最终代理对象不是同一个对象 → 出 Bug！
💡 三级缓存的精妙之处
  • 三级缓存存的是 Lambda 工厂函数，而不是对象本身
  • 工厂函数内部调用了 getEarlyBeanReference()，会触发 BeanPostProcessor 提前生成 AOP 代理
  • 所以 B 拿到的是 已经代理过的 A 半成品，和最终 A 是同一个对象
  • 这就是三级缓存存在的唯一原因——延迟执行 BeanPostProcessor

总结

问题答案
一级缓存存成品 Bean，对外提供 getBean()
二级缓存存半成品 Bean（可能已代理），解决循环依赖中"引用同一个对象"的问题
三级缓存存 Lambda 工厂，解决 AOP 代理场景下"提前暴露代理对象"的问题
为什么不能只用二级？普通对象可以，但 AOP 代理对象必须在 Bean 初始化完成前就提前生成代理，三级缓存的 Lambda 做到了"延迟执行 BeanPostProcessor"
核心源码类DefaultSingletonBeanRegistry
核心方法getSingleton(String, boolean) 和 doCreateBean()
📌 一句话总结
三级缓存是为了在"对象还没完全初始化好"的时候，就能提前生成一个正确的代理引用给别人用，确保循环依赖 + AOP 同时存在时不出 Bug。

Spring Boot 的自动装配

启动入口

SpringApplication.run() 内部做了两件大事：

  1. 推断应用类型：是 Servlet Web、Reactive Web 还是普通 Java 应用
  2. 启动 Spring 容器：最终调用 ApplicationContext.refresh()

@SpringBootApplication 的三合一

@SpringBootConfiguration
本质是 @Configuration
标注这是一个配置类
@EnableAutoConfiguration
自动装配的灵魂
触发自动配置机制
@ComponentScan
包扫描
扫描当前包及其子包

自动装配的核心机制（四步走）

第一步：@EnableAutoConfiguration 触发

AutoConfigurationImportSelector 是一个 ImportSelector，它会被 Spring 在解析 @Import 时回调。

第二步：读取 AutoConfiguration.imports

// Spring Boot 3.x 自动配置类列表
// resources/META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
org.springframework.boot.autoconfigure.web.servlet.WebMvcAutoConfiguration
org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration
org.springframework.boot.autoconfigure.orm.jpa.HibernateJpaAutoConfiguration
// ... 更多自动配置类

第三步：条件注解过滤

@Configuration
@ConditionalOnClass({ Servlet.class, DispatcherServlet.class })
@ConditionalOnWebApplication(type = Type.SERVLET)
@ConditionalOnMissingBean(DispatcherServlet.class)
public class DispatcherServletAutoConfiguration {
    // 只有满足所有条件，这个配置才会生效
}

条件注解的本质是实现了 Condition 接口，核心方法是 matches()，返回 true 才让配置生效。

第四步：生成 Bean 定义，注册到容器

通过条件过滤的配置类中，@Bean 方法会被解析成 BeanDefinition，最终注册到 DefaultListableBeanFactory 的 beanDefinitionMap 中。

🌟 自动做了什么
它只能自动配置"通用的、标准的"东西，比如：
  • 类路径有 spring-boot-starter-web → 自动配置 Tomcat、DispatcherServlet、Jackson
  • 类路径有 mybatis-spring-boot-starter → 自动配置 SqlSessionFactory、MapperScanner
  • 类路径有 DataSource → 自动配置 DataSource、JdbcTemplate

为什么还需要手动 @Bean？

🎯 场景 1：自定义业务逻辑
Spring Boot 不可能知道你的业务需要什么拦截器，它只能配置通用的。
🎯 场景 2：修复版本兼容性问题
Spring Boot 4 默认使用 Jackson 3.x，但项目可能使用 Jackson 2.x，必须显式配置。
🎯 场景 3：自定义第三方库参数
Spring Boot 不知道你的 XXL-JOB 的 appName、port 是什么，必须在 @Bean 方法里手动组装。
📌 核心原则
自动配置解决"通用"问题，手动 @Bean 解决"业务特殊"问题。

两种发现 @Configuration 的机制

📂 @ComponentScan 主启动类所在包及其子包自动扫描

适用：有 @SpringBootApplication 的模块
com.isnowsoft.lxb.daemon
📦 AutoConfiguration.imports SPI 机制，显式注册自动配置类

适用：公共模块（无启动类）
lxb-common-feign
💡 为什么需要两种机制？
@ComponentScan 扫不到依赖包里的类（包名不同），所以公共模块必须通过 AutoConfiguration.imports 显式注册。例如：lxb-common-feign 被 lxb-gateway 引入，如果不通过 SPI，@ComponentScan 根本扫不到它。

用一个完整的请求链路串联起来

🌐 客户端请求 → 🔄 DispatcherServlet → 🔀 HandlerMapping → 🎯 HandlerAdapter → ⚡ Controller → 📦 Service → 🗄️ Repository → ✅ 响应返回

这一整条链路，没有任何地方需要你手动 new，全部由容器管理和注入。
