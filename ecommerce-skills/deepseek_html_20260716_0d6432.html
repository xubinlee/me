微服务架构篇

Q1：跨境电商为什么要做微服务拆分？单体架构撑不住吗？

业务视角

  • 跨境电商业务复杂度远超国内电商：多国站点、多币种、多语言、多物流链路、多合规体系，单体架构的代码耦合度会随国家站点增加呈指数级膨胀
  • 各国家站点迭代节奏不同（俄罗斯大促是11月、东南亚大促是6月斋月后），单体架构无法按站点独立发布
  • 跨境业务对可用性要求更高（跨境网络不稳定），需要核心链路（下单/支付）与非核心链路（推荐/评价）隔离，避免非核心故障拖垮核心

技术视角

  • 单体架构痛点量化：代码量 50W+ 行 → 编译部署 30min+ → 全量发布风险高 → 一个模块 OOM 整机挂掉
  • 微服务拆分后：订单/库存/物流/支付/商品/用户 6 大核心域独立部署，提高核心链路可用性
  • 拆分原则：DDD 领域驱动，按业务边界而非技术层拆分，避免分布式单体
💡 加分点
"我们拆分时遵循两个原则：一是业务闭环（每个服务能独立完成一个业务场景），二是数据自治（每个服务拥有自己的数据库，不跨库 JOIN）。拆分后订单服务独立数据库，库存服务独立数据库，通过领域事件（RocketMQ）实现最终一致性，而非分布式事务。"

Q2：微服务拆分后，分布式事务怎么处理？Seata 还是 MQ？

维度 Seata AT 模式 RocketMQ 事务消息
一致性 强一致（分支事务全部提交或回滚） 最终一致（消息确认+补偿）
性能 全局锁持有时间长，并发度低 无全局锁，并发度高
侵入性 需要 undo_log 表，对业务有侵入 仅需实现本地事务+回查接口
适用场景 资金类强一致要求 电商类最终一致即可
跨境场景适配 跨境网络延迟大，全局锁持有时间更长 天然适配异步场景，不受跨境网络影响

跨境电商最佳实践：下单流程

1 订单服务：创建订单（本地事务） → 2 发送半消息 → 3 库存服务：扣减库存（本地事务） → 4 消费消息 → 5 库存扣减失败 → 订单回滚（RocketMQ 回查机制）
🚀 为什么不用 Seata？
  • 下单链路涉及订单 + 库存 + 营销 3 个服务，跨境网络 RT 200ms+
  • Seata 全局锁持有时间 = 本地事务时间 + 网络 RT × 2，跨境场景下全局锁持有时间翻倍
  • 大促场景下全局锁竞争激烈，TPS 下降明显
  • RocketMQ 事务消息：本地事务 + 异步消息，无全局锁，TPS 不受跨境网络影响
💡 加分点
"我们选择 RocketMQ 事务消息而非 Seata，核心原因是跨境场景下网络 RT 高（200ms+），Seata 全局锁持有时间长导致并发度低。但资金类场景（退款/结算）我们用本地消息表 + 定时补偿，保证资金强一致。"

Q3：服务间调用用什么？Feign 还是 gRPC？跨境网络延迟怎么优化？

方案选型

  • 内部服务间调用：OpenFeign（Spring Cloud 生态一致性好，开发效率高）
  • 高频核心链路（库存查询/价格查询）：gRPC（Protobuf 序列化体积小、HTTP/2 多路复用，RT 比 Feign 低 30%+）
  • 跨境网关-海外服务：gRPC（减少网络传输量）

跨境网络延迟优化策略

策略1：就近部署 东南亚服务部署在新加坡 AWS，俄罗斯服务部署在莫斯科 Yandex Cloud，国内管理后台部署在深圳阿里云。用户请求就近接入，RT 降低 60%。
策略2：异步化 同步调用→异步消息：订单创建后异步通知库存/物流，核心链路 RT 从 800ms 降至 200ms。本地缓存+异步刷新：商品信息/汇率等读多写少数据。
策略3：超时与重试 核心链路超时 3s，非核心 5s。重试策略：指数退避（1s/2s/4s），最大 3 次。幂等保障：每次请求携带唯一 requestId，服务端去重。
策略4：数据压缩 gRPC 默认 Protobuf，比 JSON 体积小 3-5 倍。大列表查询（如商品列表）启用 Gzip 压缩，传输量减少 70%。
💡 加分点
"跨境网络最大的问题不是延迟，而是不确定性——同一链路 RT 可能从 50ms 跳到 500ms。所以我们设计了自适应超时策略：基于滑动窗口统计最近 100 次调用的 P99 RT，动态调整超时阈值。正常时超时 3s，网络抖动时自动放宽到 5s，避免误触发熔断。"

高并发篇

Q4：二级库存机制具体怎么实现的？Redis 预扣和 DB 扣减如何保证一致？

Redis Lua 原子扣减脚本

-- KEYS[1]: 库存 key (inventory:sku:{skuId}:warehouse:{warehouseId})
-- ARGV[1]: 扣减数量
-- ARGV[2]: 订单号（用于幂等去重）

local stockKey = KEYS[1]
local deductQty = tonumber(ARGV[1])
local orderId = ARGV[2]

-- 幂等检查：是否已扣减过
if redis.call('SISMEMBER', stockKey .. ':orders', orderId) == 1 then
    return 0 -- 已扣减，直接返回
end

-- 库存检查
local currentStock = tonumber(redis.call('GET', stockKey) or '0')
if currentStock < deductQty then
    return -1 -- 库存不足
end

-- 扣减库存 + 记录订单号（幂等）
redis.call('DECRBY', stockKey, deductQty)
redis.call('SADD', stockKey .. ':orders', orderId)
redis.call('EXPIRE', stockKey .. ':orders', 86400) -- 24h 过期

return 1 -- 扣减成功

一致性保障三层机制

层级机制作用
第一层 RocketMQ 事务消息 Redis 扣减成功 → 发半消息 → DB 扣减 → 消息确认；DB 扣减失败 → 消息回滚 → Redis 归还
第二层 延迟消息补偿 下单后 30min 未支付 → 延迟消息触发 → Redis 归还 + DB 归还（防止库存永久锁定）
第三层 T+1 库存对账 每日凌晨对比 Redis 与 DB 库存快照，差异自动告警+工单，兜底保障
💡 加分点
"为什么不用 Redis + Lua + DB 事务的同步方案？因为 DB 行级锁（SELECT FOR UPDATE）在高并发下持有锁时间长，TPS 上不去。我们选择 Redis 异步预扣 + MQ 最终一致，Redis 层 TPS 可达 10W+，DB 层通过 MQ 削峰后 TPS 只需支撑 5000+，大促零超卖。"

Q5：库存预扣后用户不支付怎么办？超卖和少卖怎么防？

超时未支付库存释放机制

1 下单 → 2 支付等待（30min） → 3 超时未支付 → 4 库存释放
  • 下单成功后发送延迟消息（orderId, skuId, qty, warehouseId）
  • 30min 后消费者收到消息，查询订单状态：
    • 未支付 → 释放 Redis 库存 + 发 MQ 通知 DB 归还库存 + 取消订单
    • 已支付 → 忽略（幂等）
  • 释放库存也需要幂等（用 orderId 去重），避免重复释放

超卖防护

  • Redis 预扣时 Lua 原子操作保证不超卖
  • DB 层行级锁（SELECT...FOR UPDATE）作为最终兜底
  • 双重保障：即使 Redis 异常，DB 行级锁也能防止超卖

少卖防护（库存卖不完）

  • 问题：Redis 预扣后用户不支付，库存被锁定但实际可售
  • 解决：设置库存锁定超时时间（30min），超时自动释放
  • 优化：大促场景缩短锁定时间至 15min，提高库存流转效率
💡 加分点
"少卖比超卖更隐蔽但同样影响营收。我们做了一个优化：大促期间库存锁定时间从 30min 缩短至 15min，同时在商品详情页展示库存紧张标签（库存 < 10 件时），制造紧迫感加速用户支付决策，支付转化率提升了 12%。"

多仓库存篇

多仓库存调度与扣减流程

选仓路由流程

1 用户下单 → 2 选仓路由引擎 → 3 确定发货仓 → 4 扣减对应仓库存

选仓路由规则（优先级从高到低）

  1. 库存充足：该仓有库存且满足订单数量
  2. 距离最近：物流时效最短（国内仓 → 海外仓 → 国内直发）
  3. 成本最低：物流成本 + 关税成本综合最低
  4. 同国优先：优先从用户所在国家的海外仓发货（清关快）
📦 具体策略示例
  • 越南用户下单 → 优先越南海外仓 → 次选东南亚其他海外仓 → 兜底国内直发
  • 俄罗斯用户下单 → 优先俄罗斯海外仓 → 兜底国内直发
  • 库存不足时 → 拆单（有库存的仓先发）+ 缺货部分显示"到货后发货"

多仓库存扣减流程

  1. 查询分仓库存（Redis 分仓库存 key: inventory:sku:{skuId}:warehouse:{warehouseId}）
  2. 选仓路由确定发货仓
  3. 扣减指定仓库存（Lua 原子操作，key 包含 warehouseId）
  4. 如果需要拆单：
    • 订单拆分为多个子订单
    • 每个子订单对应一个发货仓
    • 分别扣减各仓库存
  5. MQ 异步同步 DB
💡 加分点
"选仓路由最大的挑战是库存快照不一致——用户看到有库存，下单时可能已被其他用户抢走。我们用 Redis Lua 原子操作做'查询 + 扣减'一体化，不是先查再扣，而是 Lua 脚本里判断库存充足直接扣减，保证原子性。扣减失败时自动尝试下一个仓，对用户透明。"
