微服务架构篇

Q：跨境电商为什么要做微服务拆分？拆分的边界怎么定？

业务视角

  • 跨境电商业务复杂度远超国内电商：多国站点、多币种、多语言、多物流链路、多合规体系，单体架构的代码耦合度会随国家站点增加呈指数级膨胀
  • 跨境电商业务链路长（商品-订单-支付-清关-物流-仓储-售后），单体架构下任何一个模块出问题都会拖垮整个系统
  • 各国家站点迭代节奏不同（俄罗斯大促是 11 月、东南亚大促是 6 月斋月后），单体架构无法按站点独立发布
  • 跨境业务对可用性要求更高（跨境网络不稳定），需要核心链路（下单/支付）与非核心链路（推荐/评价）隔离
  • 跨境场景下不同业务模块的流量特征差异大：商品浏览 QPS 高但可降级、支付 QPS 低但不可丢失、物流查询 QPS 中等但 RT 敏感

技术视角

  • 拆分边界：遵循 DDD 领域边界，核心划分 6 大域
🛍️ 商品域
商品中心 · 类目中心 · SKU 映射
多国合规审核 · 多语言商品信息
📦 订单域
订单中心 · 售后中心
多国履约规则 · COD 确认机制
📊 库存域
库存中心 · 调拨中心
多仓共享 · 跨境调拨 · 在途库存
💳 支付域
支付网关 · 结算中心
多国本地支付 · 汇率转换 · 跨境结算
🚚 物流域
物流调度 · 轨迹追踪
头程/尾程分离 · 清关节点 · 多物流商适配
👤 用户域
用户中心 · 认证中心
多国 SSO · VK/Yandex 等本地登录
  • 服务间通信：同域内同步调用（Feign + Sentinel），跨域异步解耦（RocketMQ 事件驱动）
  • 单体痛点量化：代码量 50W+ 行 → 编译部署 30min+ → 全量发布风险高 → 一个模块 OOM 整机挂掉
💡 加分点
"我们拆分时遵循两个原则：一是业务闭环（每个服务能独立完成一个业务场景），二是数据自治（每个服务拥有自己的数据库，不跨库 JOIN）。拆分后订单服务独立数据库，库存服务独立数据库，通过领域事件（RocketMQ）实现最终一致性，而非分布式事务。"

Q：微服务拆分后，分布式事务怎么处理？Seata 还是 MQ？

维度Seata AT 模式RocketMQ 事务消息
一致性强一致（分支事务全部提交或回滚）最终一致（消息确认 + 补偿）
性能全局锁持有时间长，并发度低无全局锁，并发度高
侵入性需要 undo_log 表，对业务有侵入仅需实现本地事务 + 回查接口
适用场景资金类强一致要求电商类最终一致即可
跨境适配跨境网络延迟大，全局锁持有时间更长天然适配异步场景，不受跨境网络影响
1 订单服务：创建订单（本地事务） → 2 发送半消息 → 3 库存服务：扣减库存（本地事务） → 4 消费消息 → 5 库存扣减失败 → 订单回滚（RocketMQ 回查机制）
🚀 为什么不用 Seata？
  • 下单链路涉及订单 + 库存 + 营销 3 个服务，跨境网络 RT 200ms+
  • Seata 全局锁持有时间 = 本地事务时间 + 网络 RT × 2，跨境场景下全局锁持有时间翻倍
  • 大促场景下全局锁竞争激烈，TPS 下降明显
  • RocketMQ 事务消息：本地事务 + 异步消息，无全局锁，TPS 不受跨境网络影响
💡 加分点
"我们选择 RocketMQ 事务消息而非 Seata，核心原因是跨境场景下网络 RT 高（200ms+），Seata 全局锁持有时间长导致并发度低。但资金类场景（退款/结算）我们用本地消息表 + 定时补偿，保证资金强一致。"

Q：服务间调用用什么？Feign 还是 gRPC？跨境网络延迟怎么优化？

  • 内部服务间调用：OpenFeign（Spring Cloud 生态一致性好，开发效率高）
  • 高频核心链路（库存查询/价格查询）：gRPC（Protobuf 序列化体积小、HTTP/2 多路复用，RT 比 Feign 低 30%+）
  • 跨境网关-海外服务：gRPC（减少网络传输量）
策略1：就近部署东南亚服务部署在新加坡 AWS，俄罗斯服务部署在莫斯科 Yandex Cloud，国内管理后台部署在深圳阿里云。用户请求就近接入，RT 降低 60%。
策略2：异步化同步调用→异步消息：订单创建后异步通知库存/物流，核心链路 RT 从 800ms 降至 200ms。本地缓存 + 异步刷新：商品信息/汇率等读多写少数据。
策略3：超时与重试核心链路超时 3s，非核心 5s。重试策略：指数退避（1s/2s/4s），最大 3 次。幂等保障：每次请求携带唯一 requestId，服务端去重。
策略4：数据压缩gRPC 默认 Protobuf，比 JSON 体积小 3-5 倍。大列表查询（如商品列表）启用 Gzip 压缩，传输量减少 70%。
💡 加分点
"跨境网络最大的问题不是延迟，而是不确定性——同一链路 RT 可能从 50ms 跳到 500ms。所以我们设计了自适应超时策略：基于滑动窗口统计最近 100 次调用的 P99 RT，动态调整超时阈值。正常时超时 3s，网络抖动时自动放宽到 5s，避免误触发熔断。"

高并发篇

Q：二级库存机制具体怎么实现的？Redis 预扣和 DB 扣减如何保证一致？

Redis 预扣库存 Lua 脚本核心逻辑

-- KEYS[1] = 可售库存 Key
-- KEYS[2] = 锁定库存 Key
-- KEYS[3] = 幂等记录 Key (inventory:lock:orders:{orderId})
-- ARGV[1] = 扣减数量
-- ARGV[2] = 订单号
-- ARGV[3] = 锁定过期时间(秒)

-- 1. 幂等检查（先检查是否已处理）
if redis.call('EXISTS', KEYS[3]) == 1 then
    return 1 -- 已扣减，直接返回成功（防止业务重试报错）
end

local available = tonumber(redis.call('GET', KEYS[1]) or '0')
local deductQty = tonumber(ARGV[1])

if available >= deductQty then
    -- 原子扣减
    redis.call('DECRBY', KEYS[1], deductQty)
    redis.call('INCRBY', KEYS[2], deductQty)

    -- 记录锁定明细（Hash 存主数据）
    local detail = cjson.encode({
        qty = deductQty,
        time = ARGV[3],
        sku = KEYS[1]
    })
    redis.call('HSET', 'inventory:lock:detail', ARGV[2], detail)

    -- 设置幂等标记（过期时间略大于锁定时间，防止明细未清理时重复扣）
    redis.call('SETEX', KEYS[3], tonumber(ARGV[3]) + 60, '1')

    return 1
else
    return 0
end

DB 行级锁扣减

-- SELECT FOR UPDATE 行级锁，保证 DB 层库存一致性
-- 在事务中执行：
SELECT available_qty FROM inventory
WHERE sku_id = #{skuId} AND warehouse_id = #{warehouseId}
FOR UPDATE;

-- 判断 available_qty >= deductQty 后执行：
UPDATE inventory
SET available_qty = available_qty - #{deductQty},
    locked_qty = locked_qty + #{deductQty}
WHERE sku_id = #{skuId} AND warehouse_id = #{warehouseId}
  AND available_qty >= #{deductQty};

RocketMQ 事务消息保障最终一致性

@Component
public class InventoryTransactionListener implements TransactionListener {

    @Override
    public LocalTransactionState executeLocalTransaction(Message msg, Object arg) {
        try {
            // 执行 DB 库存扣减（SELECT FOR UPDATE + UPDATE）
            inventoryService.deductDB(skuId, warehouseId, qty);
            return LocalTransactionState.COMMIT_MESSAGE;
        } catch (Exception e) {
            return LocalTransactionState.ROLLBACK_MESSAGE;
        }
    }

    @Override
    public LocalTransactionState checkLocalTransaction(MessageExt msg) {
        // 事务回查：查询 DB 库存扣减记录是否存在
        boolean deducted = inventoryService.checkDeductRecord(orderId);
        return deducted ? LocalTransactionState.COMMIT_MESSAGE
                       : LocalTransactionState.ROLLBACK_MESSAGE;
    }
}

超时未支付库存归还

// 延迟消息消费：30分钟超时未支付 → 归还库存
@RocketMQMessageListener(topic = "ORDER_TIMEOUT_CHECK",
                          consumerGroup = "inventory-timeout-group")
public class OrderTimeoutConsumer implements RocketMQListener<OrderTimeoutEvent> {

    @Override
    public void onMessage(OrderTimeoutEvent event) {
        // 1. 查询订单状态，确认确实未支付
        Order order = orderService.getById(event.getOrderId());
        if (order.getStatus() != OrderStatus.UNPAID) {
            return; // 已支付，不归还
        }

        // 2. 归还 Redis 库存
        redisTemplate.opsForValue().increment(
            "inventory:available:" + skuId + ":" + warehouseId, qty);
        redisTemplate.opsForValue().decrement(
            "inventory:locked:" + skuId + ":" + warehouseId, qty);

        // 3. 归还 DB 库存
        inventoryService.releaseInventory(skuId, warehouseId, qty);

        // 4. 取消订单
        orderService.cancelOrder(event.getOrderId());
    }
}

一致性保障四层机制

层级机制作用
第一层 Redis 抗流量 Lua 原子扣减，10W+ TPS 扛住大促峰值流量
第二层 RocketMQ 事务消息 Redis 扣减成功 → 发半消息 → DB 扣减 → 消息确认；DB 扣减失败 → 消息回滚 → Redis 归还
第三层 延迟消息补偿 下单后 30min 未支付 → 延迟消息触发 → Redis 归还 + DB 归还（防止库存永久锁定）
第四层 T+1 库存对账 每日凌晨对比 Redis 与 DB 库存快照，差异自动告警 + 工单，兜底保障
💡 加分点
"纯 Redis 方案最大的风险是数据丢失——Redis 主从切换期间（哨兵选举约 10-30s），这段时间内的库存扣减数据可能丢失。跨境大促场景下，10 秒可能产生数千笔订单，如果这些库存扣减丢失，就会导致超卖。所以我们的设计原则是：Redis 打并发流量，DB 保数据一致，MQ 保最终一致，T+1 对账兜底——四层保障，任何一层出问题都有下一层兜底。"

Q：库存预扣后用户不支付怎么办？超卖和少卖怎么防？

超时未支付库存释放机制

1 下单 → 2 支付等待（30min） → 3 超时未支付 → 4 库存释放

实现方式：RocketMQ 延迟消息（Level 16 = 30min）

  1. 下单成功后发送延迟消息（orderId, skuId, qty, warehouseId）
  2. 30min 后消费者收到消息
  3. 查询订单状态：
    • 未支付 → 释放 Redis 库存 + 发 MQ 通知 DB 归还库存 + 取消订单
    • 已支付 → 忽略（幂等）
  4. 注意：释放库存也需要幂等（用 orderId 去重），避免重复释放

超卖防护

  • Redis 预扣时 Lua 原子操作保证不超卖
  • DB 层行级锁（SELECT ... FOR UPDATE）作为最终兜底
  • 双重保障：即使 Redis 异常，DB 行级锁也能防止超卖

少卖防护（库存卖不完）

  • 问题：Redis 预扣后用户不支付，库存被锁定但实际可售
  • 解决：设置库存锁定超时时间（30min），超时自动释放
  • 优化：大促场景缩短锁定时间至 15min，提高库存流转效率
💡 加分点
"少卖比超卖更隐蔽但同样影响营收。我们做了一个优化：大促期间库存锁定时间从 30min 缩短至 15min，同时在商品详情页展示'库存紧张'标签（库存 < 10 件时），制造紧迫感加速用户支付决策，支付转化率提升了 12%。"

Q：多仓场景下库存怎么扣？用户下单时怎么选仓？

选仓路由策略

1 用户下单 → 2 选仓路由引擎 → 3 确定发货仓 → 4 扣减对应仓库存

选仓路由规则（优先级从高到低）

  1. 库存充足：该仓有库存且满足订单数量
  2. 距离最近：物流时效最短（国内仓 → 海外仓 → 国内直发）
  3. 成本最低：物流成本 + 关税成本综合最低
  4. 同国优先：优先从用户所在国家的海外仓发货（清关快）

具体策略示例

  • 越南用户下单 → 优先越南海外仓 → 次选东南亚其他海外仓 → 兜底国内直发
  • 俄罗斯用户下单 → 优先俄罗斯海外仓 → 兜底国内直发
  • 库存不足时 → 拆单（有库存的仓先发）+ 缺货部分显示"到货后发货"

多仓库存扣减流程

  1. 查询分仓库存（Redis 分仓库存 key: inventory:sku:{skuId}:warehouse:{warehouseId}）
  2. 选仓路由确定发货仓
  3. 扣减指定仓库存（Lua 原子操作，key 包含 warehouseId）
  4. 如果需要拆单：
    • 订单拆分为多个子订单
    • 每个子订单对应一个发货仓
    • 分别扣减各仓库存
  5. MQ 异步同步 DB
💡 加分点
"选仓路由最大的挑战是库存快照不一致——用户看到有库存，下单时可能已被其他用户抢走。我们用 Redis Lua 原子操作做'查询 + 扣减'一体化，不是先查再扣，而是 Lua 脚本里判断库存充足直接扣减，保证原子性。扣减失败时自动尝试下一个仓，对用户透明。"

Q：统一事件总线怎么设计的？如何避免点对点对接的复杂度爆炸？

事件总线架构

┌
│                    统一事件总线 (RocketMQ)                     
│                                                             
│  领域事件定义：                                                
│  ├── OrderCreatedEvent      → 订单创建                       
│  ├── PaymentSuccessEvent    → 支付成功                       
│  ├── InventoryDeductedEvent → 库存扣减                       
│  ├── ShipmentDispatchedEvent→ 物流发货                       
│  ├── OrderDeliveredEvent    → 订单签收                       
│  └── SettlementCompletedEvent→ 结算完成                      
│                                                             
│  事件消费者：                                                 
│  ├── 库存服务 ← PaymentSuccessEvent（确认扣减）               
│  ├── 物流服务 ← PaymentSuccessEvent（创建发货单）             
│  ├── 通知服务 ← PaymentSuccessEvent（发送通知）               
│  ├── 对账服务 ← PaymentSuccessEvent（记录待对账流水）         
│  ├── 营销服务 ← OrderDeliveredEvent（发放积分/优惠券）        
│  └── BI服务  ← *(所有事件)（实时数据同步）                    
└

消息可靠性保障（三层机制）

1. 生产者端——确保消息发出
   ├── 同步发送 + 重试（3次）
   ├── 本地消息表（Outbox Pattern）：业务操作+消息记录在同一事务中
   └── 定时任务扫描未发送消息，补偿发送

2. Broker端——确保消息存储
   ├── 同步刷盘（SYNC_FLUSH）
   ├── 主从同步复制（SYNC_MASTER）
   └── 消息持久化到磁盘

3. 消费者端——确保消息消费
   ├── 手动ACK（消费成功才确认）
   ├── 幂等消费（业务幂等表/唯一键）
   └── 死信队列（消费失败3次→DLQ→人工处理）

本地消息表（Outbox Pattern）实现

@Transactional
public void createOrder(Order order) {
    // 1. 业务操作：创建订单
    orderMapper.insert(order);
    
    // 2. 记录事件到本地消息表（同一事务）
    EventMessage event = EventMessage.builder()
        .eventId(UUID.randomUUID().toString())
        .eventType("OrderCreatedEvent")
        .payload(JsonUtils.toJson(order))
        .status("PENDING")
        .createTime(LocalDateTime.now())
        .build();
    eventMessageMapper.insert(event);
}

// 定时任务：扫描未发送消息，补偿发送
@Scheduled(fixedDelay = 5000)
public void compensateEventMessages() {
    List pendingEvents = eventMessageMapper.selectByStatus("PENDING", 100);
    for (EventMessage event : pendingEvents) {
        try {
            SendResult result = rocketMQTemplate.syncSend(
                event.getEventType(), event.getPayload()
            );
            if (result.getSendStatus() == SendStatus.SEND_OK) {
                event.setStatus("SENT");
                eventMessageMapper.updateById(event);
            }
        } catch (Exception e) {
            log.error("消息发送失败，等待下次补偿。eventId={}", event.getEventId());
        }
    }
}
❌ 点对点对接（N×M 问题）

断点续传机制

问题：同步10W条商品数据到领星，同步到5W条时网络中断，如何续传？

方案：
1. 分页同步：每页100条，记录当前页码到同步进度表
2. 每页同步成功后更新进度（原子操作）
3. 中断恢复时从进度表读取上次成功页码，继续同步
4. 幂等保障：每条数据携带唯一业务ID，领星侧按ID去重

同步进度表设计：
sync_progress:
  - id: 自增主键
  - erp_code: LINGXING/MBANG/GUANYIYUN
  - sync_type: PRODUCT/ORDER/INVENTORY
  - last_sync_page: 上次成功页码
  - last_sync_time: 上次同步时间
  - status: RUNNING/COMPLETED/FAILED
  - total_count: 总数据量
  - synced_count: 已同步数量

// 俄罗斯CDEK适配器
@Component
public class CdekLogisticsAdapter implements LogisticsAdapter {
    
    @Override
    public String getProviderCode() { return "CDEK"; }
    
    @Override
    public LogisticsTraceResponse queryTrace(String trackingNo) {
        // 1. 调用CDEK API（REST + 签名认证）
        CdekTraceResponse cdekResp = cdekClient.getTrace(trackingNo);
        // 2. 转换为统一轨迹模型
        return LogisticsTraceResponse.builder()
            .trackingNo(trackingNo)
            .status(mapStatus(cdekResp.getStatus()))  // CDEK状态 → 统一状态
            .traces(cdekResp.getEvents().stream()
                .map(this::mapTraceEvent)
                .collect(Collectors.toList()))
            .build();
    }
    
    // CDEK状态码 → 统一物流状态映射
    private LogisticsStatus mapStatus(String cdekStatus) {
        switch (cdekStatus) {
            case "CREATED": return LogisticsStatus.CREATED;
            case "IN_TRANSIT": return LogisticsStatus.FIRST_MILE_TRANSIT;
            case "CUSTOMS": return LogisticsStatus.CUSTOMS_CLEARANCE;
            case "DELIVERING": return LogisticsStatus.LAST_MILE_DELIVERY;
            case "DELIVERED": return LogisticsStatus.SIGNED;
            default: return LogisticsStatus.UNKNOWN;
        }
    }
}

// 物流商工厂
@Component
public class LogisticsAdapterFactory {
    
    @Autowired
    private List adapters;
    
    private Map adapterMap;
    
    @PostConstruct
    public void init() {
        adapterMap = adapters.stream()
            .collect(Collectors.toMap(LogisticsAdapter::getProviderCode, a -> a));
    }
    
    public LogisticsAdapter getAdapter(String providerCode) {
        LogisticsAdapter adapter = adapterMap.get(providerCode);
        if (adapter == null) {
            throw new BusinessException("不支持的物流商: " + providerCode);
        }
        return adapter;
    }
}

凭证映射引擎核心逻辑：
1. 接收业务事件（OrderSettled/RefundCompleted/InventoryOutbound等）
2. 根据事件类型+业务规则，映射为金蝶标准凭证模板
3. 填充凭证字段（科目代码、金额、币种、辅助核算项）
4. 调用金蝶API创建凭证
5. 记录凭证与业务单据的关联关系（便于追溯）

凭证映射规则示例

业务事件：OrderSettled（订单结算完成）
触发条件：订单状态=已结算 AND 支付状态=已到账

生成凭证：
借：1122-应收账款-平台（订单金额×结算汇率）
借：6603-财务费用-汇兑损益（汇率差异，可能为贷方）
贷：6001-主营业务收入（商品金额×结算汇率）
贷：2221-应交税费-增值税-销项税额（税额）
贷：6603-财务费用-手续费（支付渠道手续费）

辅助核算项：
- 应收账款：按平台维度（Shopee/Lazada/自研站）
- 主营业务收入：按国家站点+商品品类维度
- 应交税费：按国家站点维度（不同国家税率不同）
💡 加分点
"金蝶对接的核心难点是科目体系映射——不同国家站点的科目体系不同（俄罗斯用俄语科目、越南用越南语科目），但金蝶要求科目代码统一。我们设计了'业务语义→标准科目→本地科目'三级映射：业务事件先映射到标准科目（统一中文），再根据国家站点映射到本地科目。这样新增国家站点只需配置本地科目映射表，不需要改代码。"

Q：OMS+WMS+TMS 全链路履约的状态一致性如何保证？

全链路状态流转

📋 OMS 状态 → 🏭 WMS 状态 → 🚚 TMS 状态
📦 订单创建 → 📋 待发货 → 📦 创建出库单 → 📦 波次分配 → 📋 拣货
📦 打包 → 📋 称重 → ✅ 出库确认 → 📦 创建运单 → ✈️ 头程揽收
🚚 运输中 → 🛃 清关中 → ✅ 清关通过 → 🚚 尾程配送 → ✅ 已签收

一致性保障机制

机制实现方式作用
事件驱动 WMS/TMS 状态变更 → 发布领域事件 → OMS 订阅更新 解耦系统间状态同步，异步实时更新
幂等消费 每个事件携带唯一 eventId，OMS 消费前检查是否已处理 防止消息重复导致状态错乱
补偿机制 OMS 定时扫描"状态停滞"订单（如 OMS 已发货但 TMS 无运单超 2h），主动查询 WMS/TMS 补偿状态 修复因消息丢失导致的状态不一致
对账机制 T+1 全链路状态对账：OMS 订单状态 vs WMS 出库状态 vs TMS 物流状态 兜底保障，发现并修复不一致
💡 加分点
"OMS-WMS-TMS 全链路状态一致性最大的挑战是跨系统状态对齐。我们的设计原则是'OMS 为事实源，WMS/TMS 为执行源'：OMS 记录订单期望状态，WMS/TMS 反馈实际执行状态，通过事件驱动 + 定时对账双重保障。同时设计了状态停滞检测——OMS 定时扫描超过 2 小时未推进的订单，主动查询下游系统补偿状态，避免订单卡死在半路。"
