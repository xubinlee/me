Harness Engineering AI 工程实践三次范式跃迁

1. Prompt Engineering (2022-2024)

关注的是单次交互的优化——如何通过 Few-shot Learning、 Chain-of-Thought、角色设定 等技巧， 让模型在一次对话中给出更好的回答。

a. Few-shot Learning（少样本学习）

指：只给模型 极少标注样例（1~10 个左右）， 不用大规模微调，仅靠少量示例就能完成新任务。

📌 Few-shot Prompt 示例
  • 示例 1：这家奶茶超好喝 → 正面
  • 示例 2：服务很差，再也不来了 → 负面
  • 示例 3：面料粗糙不值这个价 → 负面
  • 问题：蛋糕奶油细腻甜度刚好 → ❓ 请判断

b. 思维链 CoT (Chain of Thought)

思维链 CoT 是一种提示工程技巧： 引导模型先写出完整分步推理过程，再给出最终答案，模拟人类一步步思考。

❌ 普通直接提问 （容易算错）

问：3 个苹果分给 2 个人，每人分 1 个，还剩几个？

模型直接输出：1 个

⚠️ 简单题还行，复杂数学/逻辑题容易出错

✅ CoT 思维链提问 （强制分步思考）

问：3 个苹果分给 2 个人，每人分 1 个，还剩几个？请一步步思考再回答。

1. 一共 3 个苹果
2. 2 个人每人拿走 1 个，一共分出去 2×1=2 个
3. 剩余 = 总数 − 分出去的 = 3−2=1
✅ 答案：1

c. 角色设定（Role Prompting）

在提示词开头给模型绑定固定身份、专业背景、说话风格、立场、输出规则， 约束模型的知识范围、语气、思考视角。

📋 完整角色模板包含：
  1. 身份：资深 Java 架构师 / 小学语文老师 / 专业律师
  2. 专业能力：精通 XX 领域、熟悉行业规范
  3. 输出要求：语言风格、篇幅、格式、禁止内容
  4. 约束条件：不能编造数据、通俗易懂、分点作答

d. 三者组合使用（工业级标准提示词）

实际使用中通常叠加三者，效果远高于单独使用：角色设定 + Few-shot 样例 + CoT 思维链

📝 模板结构：
  1. 角色定义（身份 + 输出约束）
  2. 任务说明
  3. 少量带推理过程的示例（Few-shot CoT）
  4. 当前待处理问题

2. Context Engineering (2025)

向前迈了一步，关注的是 "给 Agent 看什么"—— 动态构建的上下文窗口中应该填充哪些文档、对话历史、工具定义和 RAG 检索结果。

核心观点：大模型 Agent 没有长期记忆，所有它能思考、调用工具、回答问题的依据， 全部来自你塞进上下文窗口的内容。

完整上下文四大类组件

a. 对话历史

作用：让 Agent 记住上下文语境，理解多轮连续提问。

应该填充：

  • 用户所有历史提问（User）
  • Agent 上一轮完整回复（Assistant）
  • 工具调用记录（调用了什么工具、入参、返回结果）
  • 中间推理记录（CoT 思维过程、Few-shot 示例交互）
✂️ 动态裁剪规则
  • 短对话：全部保留
  • 长对话（逼近 token 上限）：
    • 保留最新 N 轮对话（滑动窗口截断）
    • 剔除无关闲聊、无效重复内容
    • 压缩长工具返回结果，只保留摘要
    • 优先保留包含关键参数、业务结论的轮次

b. 工具定义

作用：告诉 Agent 有哪些工具、怎么调用、参数格式、返回内容。

窗口内要放的完整信息：

  • 工具总说明：工具整体用途、适用场景
  • 每个工具元信息：工具名称、功能描述
  • 入参 Schema：参数名、类型、是否必填、参数含义、示例
  • 工具返回数据结构说明
  • 调用约束：什么场景禁止调用、调用次数限制
  • 输出格式要求：function call 必须是 JSON

c. RAG 检索结果

作用：给 Agent 外部私有知识，解决大模型幻觉、内部业务资料查询。

应该填充：

  • 高相似度 Top-K 分片文本（通常 3–8 条）
  • 每条文档附带元数据：文档来源、标题、章节、更新时间、可信度
  • 文档分段摘要（超长原文做切片压缩）
  • 冲突知识标注（多条文档信息矛盾时提前标注）
🔍 动态筛选逻辑
  1. 相关性过滤：低于相似度阈值的文档直接丢弃
  2. 时效性过滤：过期、作废的业务文档剔除
  3. 去重：召回重复内容只保留一份
  4. 权重排序：高相关、高权威文档放前面
  5. 容量控制：文档总 token 超标时压缩、减少召回条数

d. 静态/业务基础文档

  • 角色设定文档：Agent 身份、专业能力、回答语气、输出格式、禁止行为
  • 业务规则文档：公司流程、审批规范、数据权限、业务边界
  • Few-shot / CoT 示例文档：任务样例、带推理步骤的标准输出模板
  • 系统指令（System Prompt）：最高优先级全局约束

动态上下文窗口标准组装顺序

模型读取文本从上到下，信息权重依次降低：
  1. System 系统指令（角色 + 全局约束）
  2. 业务规则 / Few-shot/CoT 示例文档
  3. 工具定义列表（当前可用工具）
  4. RAG 检索到的知识库文档
  5. 完整对话历史（最新内容靠后）
  6. 当前用户最新提问

3. Harness Engineering (2026)

站在更高的抽象层次，不再只关注"一次对话"或"一次上下文窗口"， 而是设计 跨越多个会话、多个 Agent 角色、多个执行阶段的完整系统架构。

高抽象层：系统架构视角

a. 跨越多个会话（长期记忆体系）

🧠 分层记忆架构
  • 瞬时记忆：当前上下文窗口
  • 短期记忆：会话内持久存储（本轮聊天全程摘要、关键实体）
  • 长期记忆：向量库/数据库持久化，跨会话读取

b. 跨越多个 Agent 角色（多智能体协同）

  • 调度 Agent（中控）：分发任务、判断调用哪个子 Agent
  • 检索 Agent：专门做 RAG、文档解析、知识库查询
  • 工具执行 Agent：负责函数调用、API、数据库操作
  • 推理 Agent：复杂 CoT、逻辑演算、方案推演
  • 输出格式化 Agent：统一返回格式、校验输出合规

c. 跨越多个执行阶段（长流程任务编排）

📋 客户需求处理全流程示例
  1. 阶段 1：意图识别 & 信息补全
  2. 阶段 2：知识库检索 + 数据查询
  3. 阶段 3：多 Agent 分工计算/调研
  4. 阶段 4：方案整合、校验、反思 Self-Reflection
  5. 阶段 5：结果交付、存档、更新长期记忆

d. 完整系统级配套能力

  • 全局权限与数据隔离
  • 监控与日志（全链路 Agent 调用、token 消耗、幻觉检测）
  • 资源调度（多模型路由、上下文缓存、向量库扩容）
  • 迭代闭环（自动收集历史问答、自动优化 Agent 提示词）
  • 持久化存储（会话快照、任务中间状态、长期记忆库）

4. 复杂 Agent 任务四类失败模式

Anthropic 在其 Harness 工程博客中系统总结了 Agent 在复杂项目中的四种典型失败模式。 纯模型自驱动时，这四类问题几乎无法自愈。

⚠️ Failure Mode 1: One-shot Syndrome
试图一步到位，单窗口过载。模型最优推理习惯是一次性输出完整答案，不会主动做任务拆解。
支柱一：分层上下文 + 窗口填充管控
⚠️ Failure Mode 2: Premature Victory Declaration
过早宣布胜利。模型无法客观自评产出，只能基于文本逻辑判断"看起来写完了"。
支柱二：独立 Evaluator 评估 Agent
⚠️ Failure Mode 3: Premature Feature Completion
过早标记功能完成。模型只能阅读代码文本，无法在真实运行环境中验证。
支柱四：自动化端到端校验流程
⚠️ Failure Mode 4: Cold Start Problem
环境启动困难。上下文窗口是瞬时内存，会话销毁后全部丢失。
支柱三：持久化记忆 + 跨会话恢复

5. Harness Engineering 四大支柱

四根支柱是企业级长期运行、工程化 Agent 的标准化落地框架， 融合 Anthropic Harness 与 OpenAI Codex 大规模代码生产实战经验。

支柱一

上下文架构 Context Architecture

上下文精准按需供给，不多冗余、不少关键信息。顶层指令轻量化，分层延迟加载，动态窗口管控，稳定维持填充率在 40% 黄金区间。

支柱二

Agent 专业化 Agent Specialization

拆分专用分工 Agent，限制每个智能体工具权限，把「执行」和「评审评估」物理隔离。Planner + Generator + Evaluator 三角色分离。

支柱三

持久化记忆 Persistent Memory

任务进度、项目元数据、历史成果落地存储在文件/Git/数据库，而非临时上下文窗口。支持跨数十次会话、超长周期工程任务。

支柱四

结构化执行 Structured Execution

强制固定流水线 + 硬性质量门禁，机械化校验替代文本建议。杜绝 Agent 跳步骤、简化流程、未验证就交付。

6. 企业级项目 Harness Engineering 实战

一、核心顶层设计：四要素架构

📋 规则体系 .harness/rules/ 支柱四：定义不变约束（工程分层、编码规范、架构红线）
🛠️ 技能体系 .harness/skills/ 支柱二：标准化可复用 SOP，规划/编码/评审/测试独立
📚 知识库 wiki/ 支柱一：沉淀业务链路、中间件、隐性业务知识
🔄 变更管理 .harness/changes/ 支柱三：全流程持久化留存，跨会话恢复进度

二、中枢大脑：Application Owner Agent

整套 Harness 的统一入口、流程调度中控。遵循 Anthropic「Index & Map」设计， 控制在 400 行左右，轻量化索引而非百科全书。

五大核心模块：

  1. 角色与项目背景：明确身份、技术栈、全局硬性业务约束
  2. 配置中枢索引：结构化表格记录 Rules/Skills/Wiki/MCP 路径
  3. 七大固定核心职责：需求澄清、任务拆解、子任务分发、阶段验收等
  4. 十阶段流水线调度指令：每个阶段的准入条件、Skill、产出、门禁
  5. 硬性黑白行为约束清单：必须/禁止的行为列表

三、地基：三层分层上下文加载架构

L1 会话常驻层
Application Owner Agent + 3 份全局 Rules
永久加载
L2 阶段触发层
当前阶段 Skill + 对应分层规范
动态挂载/卸载
L3 按需查询层
Wiki 业务知识库、中间件规范
按需检索

四、结构化执行：十阶段强制流水线

1 需求分析 需求澄清
2 需求评审 最多 3 轮
3 编码实现 分层编码
4 编码评审 最多 2 轮
5 单元测试编写 改动驱动
6 单元测试评审 独立评审
7 代码推送 Git 提交
8 CI 自动化验证 编译+Linter+UT
9 部署自动化验证 端到端校验
10 最终用户确认 人工验收
🔑 关键工程设计
  • 精准失败回退路由：CI 报错→退回编码；测试不达标→退回测试编写
  • 循环迭代上限：限制评审轮次，防止死循环
  • 5 个人工介入节点：需求澄清、方案确认、编码验收、部署参数、最终交付
  • 机械化质量门禁：自定义 Linter + 结构校验 + 业务不变量校验

五、专业化 Agent 体系：Skill 技能包

两类核心角色物理隔离：

  • 生成类 Skill (Generator)：request-analysis、coding-skill、unit-test-write — 仅拥有文件读写、代码生成工具
  • 评审校验类 Skill (Evaluator)：expert-reviewer、code-review、unit-test-ci — 仅拥有读取、校验、输出缺陷报告权限

六、持久化记忆：变更管理 Changes 目录

核心文件 summary.md：全流程唯一事实源，记录每个阶段状态、评审结论、CI 指标、例外问题。

  • 每条需求独立创建隔离目录
  • 全流程文件版本留存（v1/v2/v3 不删除历史文件）
  • 构建完整审计链路 Audit Trail

7. 整体落地核心思想

  1. 完全遵循 Harness 四大支柱，全部依靠外部结构化约束、自动化反馈、持久化存储、分工式多智能体
  2. 所有隐性开发经验、架构规范、业务约束全部显性化为机器可读文件
  3. 上下文严格分层管控，稳定维持窗口填充率在安全区间
  4. 全流程可追溯、可断点续跑、自动化校验兜底
